# 프로젝트 분석

## 1. 목적

`The Thai`는 태국 마사지샵 운영 업무를 모바일에서 처리하기 위한 Android 중심 Flutter 앱이다. 핵심 업무는 다음 네 가지다.

1. 고객 등록 및 조회
2. 방문 기록과 쿠폰 잔액 관리
3. 전화 수신 시 고객 식별과 후속 액션 유도
4. 예약 등록 및 일정 관리

## 2. 기능 분석

### 고객 관리

- 고객 목록을 이름 오름차순으로 조회한다.
- 이름 또는 전화번호로 검색할 수 있다.
- 신규 고객 등록 시 이름, 전화번호, 회원 유형, 메모를 저장한다.
- 고객 상세에서 메모를 수정할 수 있다.

### 방문/쿠폰 관리

- 고객 상세 화면에서 서비스 메뉴를 선택해 방문 기록을 추가한다.
- 방문 시간에 따라 `주간` 또는 `야간`으로 자동 분류한다.
- 쿠폰 잔액이 충분하면 서비스 금액 전액을 차감한다.
- 잔액이 부족하면 가능한 금액만 차감하고 부족분 결제를 안내한다.
- 잔액이 0 이하이면 방문 이력만 저장한다.
- 고객별 쿠폰 잔액을 직접 충전할 수 있다.

### 전화 수신 연동

- 앱이 포그라운드일 때 수신 전화를 오버레이 배너로 표시한다.
- 앱이 백그라운드일 때는 시스템 알림으로 표시한다.
- 전화번호를 정규화해 고객 DB와 매칭한다.
- 통화 종료 후 예약 등록 여부를 묻고, 바로 예약 등록 화면으로 이동시킨다.
- 앱이 종료된 상태에서도 Android `BroadcastReceiver`로 수신/종료 이벤트를 기록해 복구한다.

### 예약 관리

- 날짜별 예약 목록을 조회한다.
- 예약 등록 시 기존 고객 검색 또는 미등록 고객 직접 입력을 지원한다.
- 예약 경로, 서비스, 메모를 함께 저장한다.
- 예약 상태는 `예약확정`, `취소`, `노쇼`로 관리된다.
- 노쇼 처리 시 서비스 가격을 기준으로 쿠폰 차감 여부를 선택할 수 있다.

## 3. 구조 분석

### 앱 진입과 전역 제어

- `lib/main.dart`
- Supabase 초기화
- 알림 초기화
- 전화 이벤트 수신
- 포그라운드/백그라운드 전환 감지
- 네이티브 대기 상태 복구

이 파일에 앱의 운영 흐름이 집중되어 있다. 초기 구현 속도에는 유리하지만, 앱 수명주기, 전화 이벤트, 알림 라우팅이 한 클래스에 모여 있어 복잡도가 높다.

### 데이터 계층

- `lib/services/supabase_service.dart`

`customers`, `visit_history`, `reservations`에 대한 CRUD를 직접 제공한다. 별도 repository 계층은 없고 화면이 서비스를 바로 호출하는 구조다.

장점:

- 구현이 단순하다.
- 기능 추가 속도가 빠르다.

단점:

- 화면 로직과 데이터 로직 결합이 강하다.
- 트랜잭션성 작업이 클라이언트에 흩어져 있다.
- 향후 테스트와 확장이 어려워질 수 있다.

### 화면 계층

- `lib/screens/home_screen.dart`: 고객 목록/검색
- `lib/screens/customer_add_screen.dart`: 신규 고객 등록
- `lib/screens/customer_detail_screen.dart`: 고객 상세, 메모, 방문 기록, 쿠폰 충전
- `lib/screens/reservation_screen.dart`: 날짜별 예약 목록
- `lib/screens/reservation_add_screen.dart`: 예약 등록
- `lib/screens/main_shell.dart`: 하단 탭 셸

### 네이티브 연동 계층

- `lib/services/phone_service.dart`
- `lib/services/notification_service.dart`
- `lib/services/native_call_service.dart`
- `android/app/src/main/kotlin/com/example/the_thai/PhoneStateReceiver.kt`
- `android/app/src/main/kotlin/com/example/the_thai/NotificationActionReceiver.kt`
- `android/app/src/main/kotlin/com/example/the_thai/MainActivity.kt`

Flutter 플러그인만으로 처리하기 어려운 종료 상태의 전화 이벤트를 Android 네이티브 코드로 보완한 점이 이 프로젝트의 핵심 구현 포인트다.

## 4. DB 분석

### customers

- 고객 기본 정보
- 회원 유형
- 누적 방문 수
- 주간/야간 방문 수
- 마지막 방문일
- 쿠폰 잔액
- 메모

### visit_history

- 고객별 방문 로그
- 방문 시점의 서비스명과 서비스 가격 저장
- 주간/야간 방문 구분 저장

### reservations

- 고객 연동 예약 또는 비회원 직접 예약
- 예약 날짜/시간
- 서비스명
- 유입 경로
- 상태
- 노쇼 시 사용 쿠폰 금액

## 5. 현재 문제점과 리스크

### 하드코딩된 환경값

- `lib/main.dart`에 Supabase URL과 anon key가 직접 들어 있다.
- 배포 환경 분리와 보안 관리 관점에서 `--dart-define` 또는 별도 환경 구성으로 이동하는 편이 낫다.

### 문자열 기반 상태 분기

- 회원 유형, 방문 유형, 예약 상태, 예약 경로를 모두 문자열로 직접 비교한다.
- 오타나 텍스트 변경이 곧바로 동작 오류로 이어질 수 있다.
- `enum` 또는 상수 집합으로 정리할 여지가 크다.

### 서비스 계층의 클라이언트 직접 갱신

- 방문 추가, 노쇼 차감 같은 작업은 여러 쿼리를 순차 호출한다.
- 동시성이나 중간 실패를 고려하면 RPC 또는 DB 함수로 묶는 것이 더 안전하다.

### 테스트 부족

- 자동화 테스트는 현재 매우 얕다.
- 핵심 비즈니스 로직인 전화번호 정규화, 쿠폰 차감, 예약 상태 전환은 더 직접적으로 테스트할 필요가 있다.

### 플랫폼 종속성

- 실제 운영 흐름은 Android 권한과 BroadcastReceiver에 강하게 묶여 있다.
- iOS, 웹, 데스크톱은 사실상 운영 대상이 아니다.

## 6. 우선 개선 제안

1. 상태값을 `enum` 또는 상수로 정리
2. Supabase 환경값 분리
3. 쿠폰 차감/방문 기록/노쇼 처리를 서버 측 함수로 이동
4. 전화번호 정규화와 예약 상태 전환 로직 테스트 보강
5. 매출 통계 화면 추가
